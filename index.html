The problem

False positives are why nobody reads security reports

Traditional SAST tools flag anything that resembles a pattern — including placeholder values, test fixtures, and code that's already safe. Developers learn to ignore the noise, and real issues slip through with it.

We built this the hard way. Every detector in this project went through manual verification against real code — not just unit tests. Along the way we found and fixed the exact failure modes that make other scanners exhausting to use: SQL injection missed via .format() and %s, secrets buried in dict literals, and placeholder values wrongly flagged as critical.

The result is a detector that stays quiet on clean code and specific when it isn't.

How it works

Three tools, one pipeline

Each piece does one job and does it independently — use the CLI on its own, or let the Action run it automatically on every pull request.

01 · PARSE + DETECT

Core SDK

Parses Python and JavaScript with tree-sitter, then runs pattern-based detectors for secrets, injection, and dangerous functions.

↳ ai-code-security-reviewer
02 · SCAN + EXPLAIN

CLI

Scans a file or folder from your terminal. Outputs text, JSON, or SARIF — with your own LLM key for contextual advice, or none at all.

↳ ai-code-security-reviewer-cli
03 · AUTOMATE

GitHub Action

Drops into any workflow. Comments the summary on your PR and uploads SARIF straight to GitHub Code Scanning.

↳ ai-code-security-reviewer-action

Get started

Two minutes to your first scan

Run it locally right now, or wire it into CI so it runs on every pull request automatically.

CLI — run locally 
# install
pip install git+https://github.com/nrdiiin/ai-code-security-reviewer-cli.git

# scan your project
aicsr scan . --format text
GitHub Action — runs on every PR 
- uses: nrdiiin/ai-code-security-reviewer-action@main
  with:
    path: '.'
    fail-on: 'high'
